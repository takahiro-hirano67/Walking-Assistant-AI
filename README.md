# Walking-Assistant-AI

**白杖 × AI 歩行支援デバイス - 世界を探索する楽しさを提供する**

視覚障がい者向けの白杖装着型AIデバイスのプロトタイプ実装です。白杖に取り付けたカメラで周囲を撮影し、AIが生成した説明を音声で伝えることで、「風景を知る喜び」を提供します。

## コンセプト

### 既存の歩行支援の課題

現在、視覚障がい者向けの歩行支援技術として以下のようなものがあります：

- 自律型ナビゲーションロボット（実証実験段階、一般販売なし）
- メガネ装着型/ペン型デバイス（価格：約30万円）
- スマートフォンアプリ（月額1,000円、スマホのリソースを占有）

これらは主に「安全な移動」に焦点を当てていますが、**「風景を知りたい」「周囲を探索したい」という体験的なニーズ**には十分に応えられていません。

### このプロジェクトが提供する価値

> **「普段歩いている道。本当はどんな景色が広がっているのだろう？」**

このデバイスは、以下の価値を提供します：

- **手頃な価格**（直接材料費目標：3〜4万円）  
- **スマホのリソースを消費しない**（専用ハードウェア）  
- **使い慣れた白杖に取り付けるだけ**（導入ハードルが低い）  
- **2つのモード**（簡易モード・詳細モード）で状況に応じた使い分けが可能

## 主な機能

### 1. 簡易モード（クイック説明）

- ボタン一つで撮影 → 2〜3文の簡潔な説明を音声で提供
- 処理時間: 約3〜5秒
- 用途: 素早く周囲の概要を把握したい場面

**出力例 （テスト時より）:**

> 「あなたは今、デスクに座ってパソコンを操作しています。左手でキーボードを叩き、その右隣にはマウスがあります。正面にはディスプレイがあり、その右奥にはノートパソコンが開かれた状態で置かれています。」

### 2. 詳細モード（詳細説明）

- より詳細な状況説明（空間的な位置関係、安全上の注意点を含む）
- 処理時間: 約15秒〜30秒（AI思考プロセスを含む）
- 用途: 初めての場所で周囲を詳しく知りたい場面

**出力例 (テスト時より):**

> 「この写真は部屋の中を撮影したものです。床はフローリングで、部屋全体に物がたくさん置かれていますので、歩かれる際は足元に注意が必要です。まず、写真の左手前には、青と白のストライプ模様のクッションか座布団のようなものが見えます...」

### 3. 連打による中断機能

- 処理中に再度ボタンを押すと、音声を即座に停止
- 誤操作や「もういい」という判断に柔軟に対応

### 4. バッテリー監視（実装済み、現在は無効化）

- UPS（無停電電源装置）による電源管理
- バッテリー残量が15%以下になると音声で警告

## システム構成

### ハードウェア

| 部位       | 使用パーツ                              |
| ---------- | --------------------------------------- |
| **処理部** | Raspberry Pi 5 (16GB)                   |
| **撮影部** | Raspberry Pi Camera Module 3            |
| **冷却部** | 公式Active Cooler                       |
| **電源部** | UPS HAT / 18650バッテリー               |
| **入力部** | 2キー USBキーボード（表面に触覚突起）   |
| **出力部** | Bluetoothイヤホン（Open JTalk音声合成） |
| **筐体**   | 3Dプリント製カスタムケース              |

### ソフトウェア/ライブラリ等

| 項目               | 技術                                        |
| ------------------ | ------------------------------------------- |
| **LLM**            | Gemini 2.5 Flash (Image-to-Text)            |
| **音声合成**       | Open JTalk（女性ボイス「メイ」）            |
| **効果音/再生**    | pygame                                      |
| **カメラ制御**     | picamera2                                   |
| **マルチスレッド** | threading（ボタン入力の非ブロッキング処理） |
| **入力制御**       | keyboard（USB キーボード入力）              |

### 処理フロー

```
[ボタン押下]
    ↓
[カメラ撮影] (1024×768 JPEG)
    ↓
[Gemini API送信] (Image → Text)
    ↓
[音声合成] (Open JTalk)
    ↓
[Bluetooth出力] (イヤホンで再生)
```

## 技術スタックについて

### なぜRaspberry Pi 5か

- **AI処理に十分な性能**（4コアCPU、最大16GB RAM）
  - APIのLLMを用いるのであれば低メモリで十分（今回はオーバースペックだった）
- **豊富なエコシステム**（カメラモジュール、ドキュメント、コミュニティ）
- **低消費電力**（モバイル運用に適している）
- **開発の容易さ**（Pythonで迅速にプロトタイピング可能）

### なぜGemini 2.5 Flashか

- **画像認識精度の高さ**（詳細な状況説明が可能）
- **コストの低さ**（1回あたり約0.02円）
- **2つのモード対応**
    - 簡易モード: thinking_budget=0（高速・確実性重視）
    - 詳細モード: thinking_budget=-1（動的思考・創造性重視）

## 設計思想

### 1. 「アプリの枠を消費しない」

視覚障がい者にとって、スマートフォンは生活インフラです。限られたバッテリー、メモリ、アプリの同時使用数は貴重なリソースです。

このデバイスは専用ハードウェアとして独立動作するため：

- **他のアプリと競合しない**
- **スマホのバッテリーを消費しない**
- **アプリ切り替えの煩雑さがない**

### 2. 「機能を絞ることで、明確な価値を提供」

既存の歩行支援アプリは「障害物検出」「経路案内」など多機能です。このデバイスは意図的に機能を限定しました：

- ❌ リアルタイム障害物検出
- ❌ 自律移動誘導
- ✅ 「立ち止まって風景を知る」という単一の、しかし深い価値

この割り切りにより：

- 技術的複雑さを抑え、実現可能性を高めた
- 既存アプリと競合せず、補完関係を築ける
- ユーザーは「これは〇〇のためのデバイス」と認識しやすい

### 3. 「体験価値の重視」

技術的な機能だけでなく、**尊厳や探索の喜びという体験価値**を重視しています。
